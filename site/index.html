Reposting, but the agent previews before it publishes.

Repost-with-agent is a preview-first reposting workflow for AI agents. Saved source → destination pairs, content-hash dedupe across runs, persistent audit logs per pair, approval-gated publish. The agent drives a logged-in browser profile you control; this repo supplies the schema, dedupe, and host integrations.

01 / How it works

Three files do the work.

pairs.json

One JSON record per saved source → destination pair. Mode (preview-only / approval-required / live-approved), policy, schedule, dedupe strategy. Edit via pair create / pair show.

posted.jsonl

Append-only history of every published item. Each line records sourceItemId, canonicalUrl, contentHash, destinationId. The dedupe layer reads this on every preview and re-checks at publish time.

audit.jsonl

Append-only log of every pair.preview, pair.publish.*, and pair lifecycle event. Useful when something looks wrong — the run history is right there in plain JSONL.

02 / Install

Clone, build, preview.

~/Projects 
# 1. Get it
git clone https://github.com/EthanSK/Repost-with-agent.git
cd Repost-with-agent
./scripts/install-for-openclaw.sh

# 2. Save your first pair (preview-only by default)
npx repost-with-agent pair create \
  --source-type linkedin-profile-activity \
  --source-url "https://www.linkedin.com/in/<you>/recent-activity/all/" \
  --destination-type x-account \
  --destination-account "@<you>"

# 3. Preview before posting anything
npx repost-with-agent pair preview linkedin-to-x

# 4. Publish only with explicit approval
npx repost-with-agent pair post linkedin-to-x --approve

The persistent browser profile must already be logged into both source and destination. The agent will not handle 2FA / CAPTCHA — it stops and asks.

cross-machine

Already paired with agent-bridge? A Claude / OpenClaw session on one machine can drive Repost-with-agent on another via bridge_send_message with a /repost preview <pair-id> message. The handler is read-only by default — no remote machine can publish on your behalf.

04 / Safety

What this refuses to do.

  • ✕ Bypass CAPTCHA, 2FA, or phone / identity checks.
  • ✕ Run anti-detection / stealth browser automation.
  • ✕ Collect passwords in chat or store credentials in the repo.
  • ✕ Publish without an explicit --approve flag from the local operator.
  • ✕ Re-publish an item whose sourceItemId, canonicalUrl, or normalized contentHash is already in posted.jsonl.

Conservative cadence and approval gates are for duplicate / spam protection — not detection evasion.
